## 프로그램매니저 컨셉

1. 시스템 목표 정의
1-1. 구축 목적

출판법인의 영업/구독/회원/책판매 활동을 촉진하기 위해,
여러 형태의 이벤트성 프로젝트(프로모션/캠페인/모집 프로젝트) 를 동시에 운영하고,
각 프로젝트별로 다음을 체계적으로 관리하는 시스템 구축

프로젝트별 목표와 보상 규칙 정의
PM/간사/팀원/관리자 권한 분리
모집/판매/실적 입력 및 검증
보상 기준 자동 계산
공지/문의/보고 게시판 운영
프로젝트별 성과 대시보드 제공
1-2. 핵심 비즈니스 특징

일반 PMS와 달리 아래 특징이 강합니다.

프로젝트가 이벤트성/기간성
프로젝트별로 조건과 보상 규칙이 다름
참여자들이 계층적으로 조직 구성됨
실적에 따라 수당/보상금/인센티브가 발생
운영상 게시판, 승인, 정산, 검증 기능이 중요

즉, 이 시스템은
“프로젝트 운영 + 프로모션 성과관리 + 보상정산” 성격을 함께 가집니다.

규칙2: 거점은 산하 전문가 수가 2명 이상이어야 함
규칙3: 판매자는 월간 판매량 1세트 이상이어야 함
규칙4: 조건 충족 시 정액보상 지급
규칙5: 판매금액 기준 비율보상 지급
규칙6: 지급기간은 3개월

즉, 이 시스템은 “To-do 관리”보다
규칙 기반 성과운영 시스템에 가깝습니다.

## 프로젝트 조건설계서

시스템명: 프로그램매니저
목적: 프로젝트별 달성조건, 자격조건, 보상조건을 구조화하여 등록·판정·정산이 가능하도록 설계

1. 문서 목적

본 문서는 프로그램매니저 시스템에서 운영되는 각 프로젝트의 조건을
일관된 방식으로 설정할 수 있도록 조건 정의 기준을 수립하는 것을 목적으로 한다.

프로젝트 조건은 단순 설명문이 아니라 시스템에서 판정 가능한 형태로 구조화되어야 하며,
이를 통해 다음을 가능하게 한다.

프로젝트별 참여 자격 정의
목표 달성 여부 자동 판정
보상 계산 자동화
프로젝트 간 조건 재사용
운영 정책 변경 시 유연한 수정
감사 및 정산 이력 관리
2. 조건 설계 기본 원칙
2-1. 조건은 설명문이 아니라 구조화 데이터로 관리한다

예를 들어 아래 문장을 그대로 저장하는 것이 아니라:

PM이 거점 12명을 확보하면 월 200만원을 3개월 지급한다

이를 다음처럼 나누어 저장해야 한다.

적용대상: PM
측정항목: 하위 거점 수
비교조건: 12 이상
집계기간: 프로젝트 전체
결과유형: 정액보상
지급금액: 2,000,000원
지급주기: 월
지급기간: 3개월
2-2. 조건은 최소 3개 레벨로 분리한다
구분	설명	예시
자격조건	역할 또는 참여 가능 조건	PM은 거점 확보 가능자
달성조건	실적 인정 조건	월 1세트 판매
보상조건	실제 지급 조건	40만원 + 매출 30% 지급
2-3. 조건과 실적과 지급은 분리한다
구분	의미
조건	무엇을 만족해야 하는가
실적	실제 발생한 데이터
지급	조건 충족 후 산출된 보상 결과

즉,

조건이 있어야 하고
실적이 들어와야 하며
판정 후 지급예정/지급확정이 생성되어야 함

12. 조건 설계 시 반드시 넣어야 할 운영 항목

12-2. 취소/환불 제외

책 판매 후 취소/반품이 있을 수 있으므로 실적 인정 기준을 명확히 해야 합니다.

12-3. 적용 시점

최초 달성일 기준
승인 완료일 기준
12-4. 지급 기준월

조건 충족월에 바로 지급할지, 익월 지급할지 정해야 합니다.

12-5. 버전 관리

운영 중 조건이 바뀌어도 과거 지급 데이터가 흔들리지 않게 버전 관리가 필요합니다.

13. 조건 설계 결론

이 시스템의 프로젝트 조건은 아래 방식으로 설계하는 것이 가장 좋습니다.

설명문 기반 운영이 아니라,
적용대상 + 측정항목 + 집계기준 + 판정기준 + 결과정의
구조로 분해하여 저장하고 관리하는 방식

즉, 실무적으로는 아래 공식으로 이해하시면 됩니다.

누가(적용대상)
무엇을(측정항목)
어떤 범위로(집계기준)
얼마나(판정기준)
달성하면 무엇을 준다(결과정의)

# 프로그램매니저 개발계획


## 1. 프로젝트 개요
- 시스템명: 프로그램매니저
- 목적: 다수의 이벤트성 프로젝트를 운영하고, 프로젝트별 조건 달성 여부에 따라 보상과 정산을 관리하는 시스템 구축
- 사업 배경: 출판법인의 책판매, 구독회원, 관리회원 운영 지원

## 2. 기술 스택
- Frontend: Vite, React, TypeScript, Tailwind CSS
- Auth: Firebase Authentication
- Database: Cloud SQL (PostgreSQL)
- Deployment: Firebase Hosting
- Backend 방향: TypeScript 기반 API/Functions 연계

## 3. 권한 체계
- 프로그램매니저(시스템관리자)
- PM
- 간사
- 프로젝트팀원

## 4. 핵심 기능 방향
- 프로젝트 생성 및 운영
- 프로젝트별 조건 설정
- 참여자 및 역할 관리
- 실적 입력/집계/승인
- 보상 계산 및 정산 관리
- 프로젝트별 게시판 운영
- 대시보드 및 검증/미리보기 기능

## 5. 조건 설계 원칙
프로젝트 조건은 설명문이 아니라 구조화된 데이터로 관리한다.

### 조건 구성 축
- 적용대상
- 측정항목
- 집계기준
- 판정기준
- 결과정의

### 조건 구분
- 자격조건
- 달성조건
- 보상조건
- 제한조건

### 설계 원칙
- 조건 / 실적 / 지급 데이터를 분리한다.
- 자연어 설명은 보여주되, 내부 저장은 구조화한다.
- 하드코딩보다 조건 빌더 방식으로 설계한다.
- 초기 버전은 단일조건 중심으로 시작하고, 이후 복합조건으로 확장한다.

## 6. 예시 조건 방향
### PM 거점 확보 보상
- 적용대상: PM
- 측정항목: 하위 거점 수
- 집계범위: 직접하위
- 판정기준: 12 이상
- 보상: 월 200만원 × 3개월

### 거점 전문가 모집 보상
- 적용대상: 거점
- 측정항목: 하위 전문가 수
- 판정기준: 2 이상
- 보상: 월 80만원 × 3개월

### 유아용 책 판매 보상
- 적용대상: 전문가
- 측정항목: 월 판매세트 수
- 판정기준: 1세트 이상
- 보상: 40만원 + 판매대금의 30%

## 7. 조건 입력 화면 방향
조건 입력 화면은 아래 구조를 기준으로 한다.
- 기본정보
- 적용대상
- 조건 빌더
- 결과 설정
- 조건 미리보기
- 검증 결과
