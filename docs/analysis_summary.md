# 프로그램매니저 분석 요약 (초기 개발 기준)

## 1) 문제 정의
- 프로젝트마다 자격/달성/보상 규칙이 다르며 변경 빈도가 높다.
- 참여자 구조가 계층형(PM > 거점 > 전문가 > 팀원)이고 프로젝트별 역할이 다를 수 있다.
- 실적 입력, 승인, 보상 계산, 지급 상태를 분리해 관리해야 한다.

## 2) 핵심 설계 원칙
- 규칙 하드코딩 금지: `조건 정의 데이터 + 판정 로직` 구조로 설계
- 권한 이원화: 전역 권한 + 프로젝트 권한 분리
- 데이터 분리: `조건` / `실적` / `계산결과` / `지급` 별도 관리
- 운영 단위 분리: 게시판/실적/정산은 프로젝트 단위로 독립

## 3) 조건입력 화면에서 필요한 도메인 축
- 기본정보: 조건명, 조건구분, 적용기간, 사용여부
- 적용대상: 대상유형(역할/개인/조직), 역할, 조직범위, 중복적용
- 조건빌더: 측정항목, 집계대상/주기, 비교연산자, 기준값, 승인/취소 필터, 유지기간
- 결과설정: 결과유형, 보상유형(정액/비율/혼합), 금액/비율, 지급주기, 지급횟수, 한도
- 우측패널: 사람이 읽는 문장 미리보기 + 검증결과

## 4) MVP 범위 (이번 개발 시작 범위)
- 관리자 조건입력 화면 1차 구현
- 조건 입력 state 모델링 + 검증/미리보기 로직
- 저장/임시저장/미리보기 액션 스켈레톤

## 5) 다음 단계
- API/DB 연결: `project_rules`, `project_rule_conditions`, `project_rule_rewards`
- 조건 버전/효력기간 처리
- 복합조건(AND/OR/선행조건) 확장
